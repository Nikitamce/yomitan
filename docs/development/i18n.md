# Internationalization (i18n)

Yomitan localizes **extension chrome** (settings, popups, modals, help text) with the standard Chromium / Firefox extension system: [`chrome.i18n`](https://developer.chrome.com/docs/extensions/reference/api/i18n) and `ext/_locales/*/messages.json`.

UI language follows the **browser UI language** (no in-app switcher). English is `default_locale` and the fallback text embedded in HTML.

Dictionary **entry content** (definitions, tags inside dictionary packages) is **not** localized by this system — only the extension UI and related catalogs.

---

## Architecture overview

```
┌─────────────────────────────────────────────────────────────┐
│  Browser UI language (e.g. ru)                               │
│       ↓                                                      │
│  chrome.i18n.getMessage(key)  ←  ext/_locales/<lang>/…       │
│       ↑                                                      │
│  i18n-util.js  ·  HTML data-i18n*  ·  JS getMessage()        │
└─────────────────────────────────────────────────────────────┘
```

| Layer                               | Role                                                                             |
| ----------------------------------- | -------------------------------------------------------------------------------- |
| `ext/_locales/en/messages.json`     | Source of truth for **keys** and English strings                                 |
| `ext/_locales/<lang>/messages.json` | Translations; **must** contain every `en` key                                    |
| `ext/js/language/i18n-util.js`      | `getMessage`, `applyI18nToDocument` / `applyI18nToRoot`, card-format name helper |
| HTML / templates                    | `data-i18n`, `data-i18n-html`, … with English fallback body text                 |
| `dev/bin/i18n-check.js`             | Key parity CI (`npm run test:i18n`)                                              |
| `.github/workflows/i18n-check.yml`  | Runs check on PRs/pushes that touch locales                                      |

### HTML attributes

| Attribute                                                                                | Effect                                                                  |
| ---------------------------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| `data-i18n="key"`                                                                        | Sets `textContent`                                                      |
| `data-i18n-html="key"`                                                                   | Sets `innerHTML` (trusted locale HTML only: links, `<em>`, `<code>`, …) |
| `data-i18n-title` / `data-i18n-placeholder` / `data-i18n-aria-label` / `data-i18n-value` | Corresponding DOM properties                                            |
| `data-i18n-args="a,b"`                                                                   | Substitution args for `$1`, `$2`, …                                     |
| `data-i18n-title-key` on `<html>`                                                        | Document title                                                          |

Templates loaded via `html-template-collection.js` run `applyI18nToRoot` on template content so modals/settings fragments are localized when cloned.

### Naming conventions in the catalog

| Prefix                                    | Typical use                                                |
| ----------------------------------------- | ---------------------------------------------------------- |
| `ui_*`                                    | Plain UI labels / short text                               |
| `html_*`                                  | Markup strings (links, inline tags)                        |
| `js_*`                                    | Strings constructed only in JavaScript                     |
| `extensionName`, `extensionDescription`, … | Manifest `__MSG_*__`                                       |
| `rec_dict_desc_*`                         | Recommended dictionary **descriptions** (see below)        |
| `actionPopup_*`, `settings_nav_*`         | Older / area-specific keys                                 |

## Recommended dictionary descriptions

The “Get recommended dictionaries…” modal lists term/kanji/frequency/… packs from a content file. Descriptions are **data**, not static HTML, so they are wired into `chrome.i18n` as dedicated keys.

### Files

| File                                                   | Role                                                                                                                             |
| ------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------- |
| `ext/data/recommended-dictionaries.json`               | Source data: name, download URL, English description, optional homepage (grouped by language code and category)                  |
| `ext/data/recommended-dictionaries-i18n-keys.json`     | Map: dictionary **display name** → message key `rec_dict_desc_…`                                                                 |
| `ext/_locales/en/messages.json`                        | English description text for each `rec_dict_desc_*` key                                                                          |
| `ext/_locales/ru/messages.json`                        | Russian (and future locales) for the same keys                                                                                   |
| `dev/bin/i18n-sync-recommended-dicts.js`               | Regenerates keys + EN entries from JSON; fills missing RU with templates / known strings without overwriting hand-edited RU      |
| `ext/js/data/recommended-dictionary-i18n.js`           | `getRecommendedDictionaryDescription(name, nameToKey, fallback)`                                                                 |
| `ext/js/pages/settings/dictionary-import-controller.js`| Loads the key map, resolves description via `getMessage` when rendering the list                                                 |

### Runtime flow

1. Fetch `recommended-dictionaries.json` and filter by the user’s **dictionary language** setting (e.g. `ja`).
2. Fetch `recommended-dictionaries-i18n-keys.json` (name → key).
3. For each list item, set  
   `description.textContent = getRecommendedDictionaryDescription(name, map, englishFallback)`.
4. `getMessage(key)` returns the string for the **browser UI language**. If missing/empty, the English string from the JSON is used.

### Key shape

Keys look like:

```text
rec_dict_desc_<slug>_<8-hex>
```

- **slug**: ASCII-ish slug from the dictionary name (max 40 chars).
- **hash**: first 8 hex chars of SHA-1 of the exact `name` string (avoids collisions for similar names, e.g. Words.hk variants).

Generation must stay consistent between `dev/bin/i18n-sync-recommended-dicts.js` (Node `crypto`) and the committed map file. Runtime **does not** re-hash; it only reads the map.

### Adding or updating a recommended dictionary

1. Edit `ext/data/recommended-dictionaries.json` (name, URLs, English `description`).
2. Run:

   ```bash
   node dev/bin/i18n-sync-recommended-dicts.js
   npm run test:i18n
   ```

3. Translate new `rec_dict_desc_*` entries in each non-English locale if auto-fill left English or a template.
4. Commit: JSON data, `recommended-dictionaries-i18n-keys.json`, and locale message updates.

The sync script:

- Creates missing EN keys from English descriptions.
- Creates missing RU keys (Wiktionary/Toki Pona templates + a small known map for popular packs).
- **Does not** overwrite existing Russian text when it already differs from English (hand translations are preserved).
- Removes obsolete `rec_dict_desc_*` keys no longer present in the source JSON.

### Counts (approximate)

- ~95 language codes in `recommended-dictionaries.json`
- ~124 unique dictionary names / description keys  
  (exact count: `node -e "console.log(Object.keys(require('./ext/data/recommended-dictionaries-i18n-keys.json')).length)"`)

## Adding a new UI string

1. Prefer a stable key under `ui_*` / `html_*` / `js_*`.
2. Add the key to **`en/messages.json`** first (`message` + short `description`).
3. Add the same key to every other locale (`ru`, …).
4. Wire the UI:
   - HTML: `data-i18n` / `data-i18n-html` with English fallback body text.
   - JS: `import {getMessage} from '../language/i18n-util.js'` then `getMessage('key')` (or with substitution array).
5. Run `npm run test:i18n`.

### Nested `data-i18n` inside `data-i18n-html`

`applyI18nToElement` does **not** re-walk children after setting `innerHTML` from a parent `data-i18n-html` key. Prefer putting the full markup (including link labels) in the locale message, rather than nesting `data-i18n` inside an `html_*` string.

### Placeholders

Chrome-style placeholders:

```json
"ui_example": {
    "message": "Using $USAGE$ of $QUOTA$",
    "placeholders": {
        "USAGE": { "content": "$1" },
        "QUOTA": { "content": "$2" }
    }
}
```

```js
getMessage("ui_example", [usageString, quotaString]);
```

For HTML that must keep live DOM nodes (e.g. storage usage spans updated by a controller), put the spans **inside** the `html_*` message string so classes survive `innerHTML` replacement.

## CI and tests

| Command               | Purpose                                                         |
| --------------------- | --------------------------------------------------------------- |
| `npm run test:i18n`   | Every non-`en` locale has all `en` keys; invalid JSON fails     |
| Workflow `i18n Check` | Same on PR/push when locales or the check script change         |

Warnings (non-fatal): messages still identical to English (possible untranslated UI), with exclusions for brands/technical tokens.

Other jobs (TS, HTML, build, Playwright) remain as in the main CI matrix.

---

## Manual QA checklist (UI language)

1. Set browser UI language to the target locale (e.g. Russian); restart if needed.
2. Load a built package or unpacked `ext/` (manifest generated by the project build).
3. Smoke-test:
   - Action popup, settings sidebar, Anki / audio / shortcuts modals
   - Firefox data-transmission consent (if Firefox)
   - Welcome / quick-start guide
   - Recommended dictionaries modal (descriptions for the active language list)
   - Permissions page (nativeMessaging, file URLs)
4. Switch browser language back to English → UI returns to English.

---

## Out of scope / non-goals

- In-app language switcher independent of browser UI language
- Translating definitions inside dictionary ZIP packages
- Localizing every OS/browser chrome string outside the extension
- Auto-translating all 100+ recommended descriptions into every future locale (sync provides EN keys; humans refine)
